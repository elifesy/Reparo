⚡ Documentation v2.1

Reparo

A full-stack device service management portal for electronics repair centers. Three portals — customer, engineer, and admin — backed by a Node.js REST API and a persistent SQLite database.

v2.0 — Theme selectorv2.0 — Multi-field searchv2.1 — Smart track searchv1.0 — Initial release
📦 Node.js 22 + Express
🗄️ SQLite (better-sqlite3)
🔐 JWT Authentication
🌐 Vanilla JS SPA

Reparo gives repair centers a complete digital workflow — from walk-in intake to device dispatch — with real-time status tracking, customer notifications, and a full audit trail.

The system is intentionally designed as a single deployable unit: one npm start launches the Express server which serves the frontend, handles all API calls, and writes to a local SQLite database. No separate build step, no separate frontend server, no external dependencies at runtime.

Request flow

Every request from the browser follows the same path through the stack:

Browser
  │
  ├── Static assets (HTML/CSS/JS)
  │   └── Express static() serves public/index.html
  │
  └── API calls (fetch('/api/...')
      │
      ├── JWT header extracted by auth middleware
      ├── Token verified (signature + expiry)
      ├── Role checked against route requirement
      │
      ├── Route handler executes
      │   └── better-sqlite3 → synchronous SQL query
      │       └── SQLite file (./db/reparo.sqlite)
      │
      └── JSON response returned to browser
ℹ️
better-sqlite3 is synchronous — no async/await on queries. This simplifies the code significantly and performs well for workloads up to several hundred concurrent connections. For higher scale, migrate to PostgreSQL with the pg package.

How auth works

Reparo uses stateless JWT authentication. When a user logs in, the server signs a token containing their user ID, email, role, and name. This token is stored in the browser's localStorage and sent as a Bearer token on every subsequent API request.

The server never stores sessions — it only validates the token's signature and expiry on each request.

Token payload (decoded)
{
  "id":    "abc123",
  "email": "[email protected]",
  "role":  "Engineer",
  "name":  "Mohammed Al-Farsi",
  "iat":   1710000000,
  "exp":   1710028800   // 8 hours
}
Authorization header format
Authorization: Bearer <token>

2FA (admin)

The admin login form accepts an optional 6-digit TOTP code. In the current implementation this is validated client-side against a fixed demo code (123456). For production, integrate a proper TOTP library such as otplib and store the TOTP secret per user in the database.

⚠️
The 2FA field is currently for demonstration only. Replace it with real TOTP validation before deploying admin accounts to production.

Reparo uses a single SQLite file at ./db/reparo.sqlite. The schema is created automatically on first run by db/setup.js. WAL mode and foreign keys are enabled.

Notifications

Notification settings are stored in the database and managed through the Admin → Settings panel. The current implementation shows toast() notifications in the UI as a placeholder. To enable real notifications in production:

  • SMS — integrate Twilio and call twilio.messages.create() after service creation.
  • Email — use SendGrid or nodemailer on status changes.
  • WhatsApp — use the Twilio WhatsApp Business API or Meta's Cloud API.

Security overview

Reparo implements several layers of security appropriate for an internal business application.

Password hashing

All passwords are hashed using bcrypt with a cost factor of 10 before storage. Plain-text passwords are never persisted or logged.

db/setup.js — password hashing
const hash = bcrypt.hashSync(password, 10);
// Verification at login:
const valid = bcrypt.compareSync(plainPassword, user.password);

JWT tokens

Tokens are signed with HS256 using the JWT_SECRET environment variable. They expire after 8 hours. The middleware validates both the signature and the expiry on every protected request.

middleware/auth.js
function auth(requiredRole) {
  return (req, res, next) => {
    const header = req.headers.authorization;
    if (!header.startsWith('Bearer '))
      return res.status(401).json({ error: 'No token' });
    const payload = jwt.verify(header.slice(7), JWT_SECRET);
    req.user = payload;
    // Role check if requiredRole provided
    next();
  };
}

CORS

The cors middleware is included and configured with default settings (allow all origins). For production, restrict to your actual domain:

server.js — restrict CORS in production
app.use(cors({
  origin: 'https://reparo.yourdomain.com',
  credentials: true
}));
🔐
Production checklist: Change JWT_SECRET, restrict CORS to your domain, use HTTPS (nginx + Let's Encrypt), and replace the 2FA demo code with real TOTP validation.

How it works

Themes work by setting a data-theme attribute on the <html> element. Each theme is a CSS block that overrides the background, text, and border CSS variables defined in :root. Semantic colours (status badges, alerts, buttons) are not overridden — they stay consistent across all themes.

CSS structure
/* Default dark theme lives in :root */
:root { --bg: #0a0c10; --text: #e2e8f0; ... }

/* Each theme overrides only surface variables */
[data-theme="light"] { --bg: #f8f9fb; --text: #1a2035; ... }
[data-theme="ocean"] { --bg: #060e1e; --accent: #38bdf8; ... }
JS — applying a theme
applyTheme('ocean');
// Sets data-theme="ocean" on <html>
// Saves 'ocean' to localStorage under 'rp_theme'
// Updates the active state in the theme picker panel

Adding a custom theme

Multiple results

When a phone number or email matches more than one service (e.g. a customer has brought in three devices), all matches are shown as a card list. The customer taps a card to expand the full detail view with the progress stepper and activity timeline below the list.

API endpoint

GET /api/services/track/search
# Search by ID
GET /api/services/track/search?q=RPR-0042

# Search by phone
GET /api/services/track/search?q=%2B971501234567

# Search by email
GET /api/services/track/search?q=ahmad%40email.com

# Response
{
  "count": 2,
  "results": [
    { "id": "RPR-0042", "status": "In Progress", ... },
    { "id": "RPR-0038", "status": "Dispatched", ... }
  ]
}
ℹ️
When count === 1, the activities array is included in the single result for the full timeline. For multi-result responses, activities are fetched individually when the customer taps a card.
ℹ️
Search is case-insensitive and uses SQL LIKE '%query%' matching. It does not support exact-match quoting or boolean operators in this version.

  • Set up daily backups of db/reparo.sqlite

Upgrading to PostgreSQL

When you outgrow SQLite (heavy write concurrency, multi-server deployments), the migration path to PostgreSQL is straightforward:

1

Replace the package

npm uninstall better-sqlite3
npm install pg
2

Update db/setup.js

Replace the better-sqlite3 connection with a pg.Pool. Convert synchronous db.prepare().run() calls to await pool.query().

3

Make route handlers async

All route handler functions will need async added and await before every query.

4

Run the schema

The CREATE TABLE statements in db/setup.js are standard SQL and work in PostgreSQL with minor type adjustments (e.g. SERIAL instead of INTEGER AUTOINCREMENT).
